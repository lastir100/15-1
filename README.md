## Задача 2

Выберите один из вариантов платформы в зависимости от задачи. Здесь нет однозначно верного ответа так как все зависит от конкретных условий: финансирование, компетенции специалистов, удобство использования, надежность, требования ИБ и законодательства, фазы луны.

Тип платформы:

- физические сервера;
- паравиртуализация;
- виртуализация уровня ОС;

Задачи:

- высоконагруженная база данных MySql, критичная к отказу;
- различные web-приложения;
- Windows-системы для использования бухгалтерским отделом;
- системы, выполняющие высокопроизводительные расчёты на GPU.

Объясните критерии выбора платформы в каждом случае.

## Решение 2
- высоконагруженная база данных MySql, критичная к отказу;
  Тип платформы: физические сервера.
  Производительность и надежность: БД критична к задержкам (latency) и требует максимального использования ресурсов (I/O, CPU Cache). Физический сервер исключает накладные расходы гипервизора, обеспечивая предсказуемую производительность.
- различные web-приложения;
  Тип платформы: виртуализация уровня ОС.
  Изоляция и плотность: Web-приложениям требуется хорошая изоляция друг от друга (для безопасности), а виртуализация уровня ОС (контейнеры) обеспечивает максимальную плотность размещения и еще высокую скорость развертывания (CI/CD).
- Windows-системы для использования бухгалтерским отделом;
  Тип платформы: паравиртуализация.
  Для Windows требуется полная эмуляция железа, которую обеспечивают VM. Для работы с десктопными ОС нужен Full Virtualization.
- системы, выполняющие высокопроизводительные расчёты на GPU.
  Тип платформы: физические сервера.
  Доступ к оборудованию (I/O): Требуется прямой доступ к ресурсам GPU без накладных расходов. Лучше всего — физический сервер. 


## Задача 3

Выберите подходящую систему управления виртуализацией для предложенного сценария. Опишите ваш выбор.

Сценарии:

1. 100 виртуальных машин на базе Linux и Windows, общие задачи, нет особых требований. Преимущественно Windows based-инфраструктура, требуется реализация программных балансировщиков нагрузки, репликации данных и автоматизированного механизма создания резервных копий.
2. Требуется наиболее производительное бесплатное open source-решение для виртуализации небольшой (20-30 серверов) инфраструктуры на базе Linux и Windows виртуальных машин.
3. Необходимо бесплатное, максимально совместимое и производительное решение для виртуализации Windows-инфраструктуры.
4. Необходимо рабочее окружение для тестирования программного продукта на нескольких дистрибутивах Linux.

## Решение 3
1. Microsoft Hyper-V. Выбор обусловлен существующей Windows-based инфраструктурой и требованиями к глубокой интеграции с продуктами Microsoft (Active Directory, Windows Server). Hyper-V бесплатен, но для управления кластером, балансировки (live migration) и продвинутых функций (репликация, бэкапы) требуется SCVMM (платный) или Windows Admin Center (базовый функционал, бесплатный).
2. Proxmox Virtual Environment (PVE). Это бесплатное, открытое и высокопроизводительное решение на базе KVM (для VM) и LXC (для контейнеров). PVE включает инструменты для централизованного управления, кластеризации, создания хранилищ (Ceph/ZFS) и базового резервного копирования. Идеально для небольших инфраструктур с ограниченным бюджетом.
3. VMware ESXi Free (только для виртуализации) или Microsoft Hyper-V Server Free. ESXi Free лишен API (нет централизованного управления) и некоторых функций, но максимально производителен и совместим. Hyper-V Server (Free) обеспечивает лучшую совместимость с Windows-VM, но управление требует сторонних инструментов или Windows Admin Center. Оба бесплатны, но Hyper-V лучше интегрируется в Windows-среду.
4. Виртуализация уровня ОС (Docker). Скорость и легковесность: Для тестирования ПО на различных версиях Linux не требуется полная эмуляция ядра. Docker/Podman предоставляет мгновенное развертывание (в секундах) легковесных окружений (контейнеров) для каждого дистрибутива Linux, используя общее ядро хоста. Это обеспечивает максимальную плотность и быструю перенастройку.


## Задача 4

Опишите возможные проблемы и недостатки гетерогенной среды виртуализации (использования нескольких систем управления виртуализацией одновременно) и что необходимо сделать для минимизации этих рисков и проблем. Если бы у вас был выбор, создавали бы вы гетерогенную среду или нет?

## Решение 4
- Проблемы и Недостатки.
Сложность управления и обучения: Требуется наличие специалистов, обладающих компетенциями во всех используемых системах. Увеличиваются затраты на обучение и поддержку.

Отсутствие централизованного мониторинга: Разные платформы имеют разные API, инструменты логирования и форматы метрик. Требуется внедрение сложной унифицированной системы мониторинга (например, на базе Prometheus/Grafana), способной агрегировать данные со всех платформ.

Несовместимость форматов VM: VM, созданная в VMware (.vmdk), не может быть напрямую запущена в Hyper-V (.vhdx) или KVM (.qcow2). Миграция (перемещение) VM между платформами требует трудоемкой конвертации.

Управление ресурсами (Silo Effect): Ресурсы (CPU, RAM, Storage) распределяются независимо в каждом "островке" виртуализации. Это приводит к неэффективному использованию ресурсов и "затовариванию" в одном кластере при дефиците в другом.

- Минимизация рисков и проблем
Использование единой платформы управления: Внедрить внешнюю, независимую от гипервизора платформу управления (например, Red Hat CloudForms, OpenNebula или OpenStack) для унификации предоставления ресурсов и мониторинга.

Стандартизация VM и контейнеров: Использовать контейнеризацию (Docker/Kubernetes) там, где это возможно. Контейнеры обеспечивают гораздо лучшую переносимость, чем VM.

Автоматизация (IaC): Максимально автоматизировать развертывание и управление VM с помощью инструментов Infrastructure as Code (IaC) (например, Terraform, Ansible). Это позволяет использовать единый код для развертывания инфраструктуры независимо от того, работает ли она на Hyper-V или KVM.

Четкое разделение задач: Использовать каждую платформу только для тех задач, в которых она незаменима (например, Hyper-V только для Windows-кластеров, KVM только для высокопроизводительных Linux-задач).

- Выбор: Создавать ли гетерогенную среду?
Нет, не создавал бы, если это возможно.

Причины:

В современных условиях, однородная среда (например, только VMware или только KVM/OpenStack) почти всегда проще, дешевле в обслуживании и эффективнее.

Современные гипервизоры (KVM, Hyper-V, VMware) обладают достаточной гибкостью и производительностью для большинства задач (Linux, Windows, контейнеры, GPU-passthrough), что устраняет необходимость в гетерогенности.

Гетерогенность оправдана только в двух случаях:

Слияние компаний: Когда две компании с разной инфраструктурой объединяются.

Специфические требования: Когда требуется функциональность, строго уникальная для одной платформы (например, особенности лицензирования или глубокая интеграция, которую не может предоставить другая система).
